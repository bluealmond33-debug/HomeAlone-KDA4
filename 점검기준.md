# retry_recommendation_tool 추가 테스트 기준

## 1. 검색 결과 경계 테스트

### 빈 검색 결과

- 입력: 유효한 캐시 없음, 검색 결과 `[]`
- 기대 결과:
  - `status == "NO_NEW_CANDIDATE"`
  - `recommendation is None`
  - 기존 추천 이력 유지

### 검색 결과가 None인 경우

- 입력: 검색 파이프라인이 `None` 반환
- 기대 결과:
  - `RetryPipelineError` 발생
  - 원본 `TypeError`가 `__cause__`에 보존
- 참고: 현재 코드에서는 `TypeError`가 직접 발생하므로 보완 필요

### Generator 순회 중 오류

- 입력: 검색 결과 generator가 순회 도중 `TimeoutError` 발생
- 기대 결과:
  - `RetryPipelineError`로 변환
  - 원본 `TimeoutError`가 `__cause__`에 보존
- 참고: 현재 코드에서는 `TimeoutError`가 직접 발생하므로 보완 필요

## 2. 조리 조건 경계값 테스트

### 인분 경계값

- `servings == 1`: 추천 허용
- `servings == 2`: 추천 허용
- `servings == 3`: 추천 제외
- `servings == 0`: Pydantic 검증 오류

### 조리 시간 경계값

- `cooking_time_minutes == 30`: 추천 허용
- `cooking_time_minutes == 31`: 추천 제외
- `cooking_time_minutes == 0`: Pydantic 검증 오류

### 난이도 조건

- `"초급"`: 추천 허용
- `"아무나"`: 추천 허용
- `"중급"` 또는 `"고급"`: 추천 제외
- 빈 문자열: Pydantic 검증 오류

## 3. 카테고리 테스트

### 특정 카테고리

- 요청 카테고리와 후보 카테고리가 같으면 추천 허용
- 서로 다르면 추천 제외

### 상관없음 카테고리

- 요청 카테고리가 `"상관없음"`이면 모든 후보 카테고리 허용
- 다른 필수 조건은 그대로 검사

### 지원하지 않는 카테고리

- `"퓨전"`, 빈 문자열, `None` 입력
- 기대 결과: Pydantic `ValidationError`

## 4. 식재료 테스트

### 정확히 일치하는 식재료

- 요청: `"김치"`
- 후보: `"김치"`
- 기대 결과: 추천 허용

### 공백과 대소문자 정규화

- 요청: `"  Egg  "`
- 후보: `"egg"`
- 기대 결과: 동일한 재료로 처리

### 부분 일치

- 요청: `"김치"`
- 후보: `"배추김치"`
- 현재 기대 결과: 불일치로 처리
- 향후 부분 일치가 필요하다면 별도의 정책 정의 필요

### 중복 식재료

- 입력: `["김치", "김치", " 계란 "]`
- 기대 결과: `["김치", "계란"]`으로 정규화

### 식재료 개수

- 유효 식재료 5개: 허용
- 유효 식재료 6개: `ValidationError`
- 빈 리스트: `ValidationError`
- 공백 문자열만 있는 리스트: `ValidationError`

## 5. URL 정규화 및 보안 테스트

### 동일 URL 정규화

다음 URL은 동일한 URL로 처리되어야 한다.

- 쿼리 문자열 포함
- fragment 포함
- 모바일 서브도메인
- 마지막 `/` 포함

기대 정규화 결과:

```text
https://www.10000recipe.com/recipe/{recipe_id}
```

### 허용하지 않는 URL

- HTTP URL
- 외부 도메인
- 사용자명 또는 비밀번호 포함 URL
- 포트 번호 포함 URL
- 숫자가 아닌 recipe ID
- `/recipe/` 형식이 아닌 경로
- 빈 문자열
- `None`

기대 결과: `ValueError` 또는 Pydantic `ValidationError`

## 6. 중복 추천 방지 테스트

### URL 중복

- 제목은 다르지만 이전 추천 URL과 동일
- 기대 결과: 후보 제외

### 메뉴명 중복

다음 메뉴명은 동일하게 정규화되어야 한다.

- `"김치볶음밥"`
- `"김치 볶음밥"`
- `"김치  볶음밥"`

기대 결과: 이전 메뉴명과 중복된 후보 제외

### 검색 결과 내부 중복

- 동일 URL 후보가 여러 개 존재
- 동일 메뉴명이 여러 개 존재
- 기대 결과: 최초 후보만 유지

## 7. 캐시 우선순위 테스트

### 유효한 캐시 존재

- 기대 결과:
  - `source == "CACHE"`
  - 검색 파이프라인 호출 안 함
  - 첫 번째 유효 캐시 후보 반환

### 캐시 후보가 조건에 맞지 않는 경우

- 기대 결과:
  - 검색 파이프라인 호출
  - 검색 후보 반환 시 `source == "SEARCH"`

### 사용한 캐시 제거

- 선택된 후보는 반환되는 `cached_candidates`에서 제거
- 선택된 후보와 동일한 메뉴명의 다른 URL 후보도 제거

## 8. 결과 순서 테스트

- 검색 결과가 랭킹 순서대로 입력되면 첫 번째 유효 후보 반환
- 첫 번째 후보가 조건에 맞지 않으면 다음 유효 후보 반환
- 중복 후보는 건너뛰고 다음 후보 검사

## 9. 잘못된 검색 데이터 테스트

### 일부 데이터가 잘못된 경우

- 잘못된 후보는 제외
- 정상 후보는 계속 처리
- 제외된 후보 수만큼 `warnings` 생성

### 모든 데이터가 잘못된 경우

- `status == "NO_NEW_CANDIDATE"`
- `recommendation is None`
- 모든 검증 오류가 `warnings`에 기록

### 허용되지 않은 추가 필드

- `RecommendationResult`에 정의되지 않은 필드 포함
- 기대 결과: 해당 후보 제외 및 warning 기록

## 10. 상태 이력 테스트

### 추천 성공

- 선택된 URL이 `previous_recipe_urls`에 추가
- 정규화된 메뉴명이 `normalized_menu_names`에 추가
- 기존 이력 순서 유지

### 추천 실패

- 기존 URL 및 메뉴명 이력 변경 없음
- `recommendation is None`
- `source is None`

### 입력 객체 변경 여부

- 함수 실행 전후 원본 요청 객체가 직접 변경되지 않아야 함

## 11. 파이프라인 전달값 테스트

검색 파이프라인에 다음 값이 정확히 전달되는지 확인한다.

- `valid_ingredients`
- `category`
- `exclude_urls`
- `exclude_menu_names`

추가 검증:

- `exclude_urls`는 `frozenset`
- `exclude_menu_names`는 `frozenset`
- URL과 메뉴명은 정규화된 값으로 전달

## 12. 최종 통과 기준

- 기존 테스트 8개와 추가 테스트 모두 통과
- 예상하지 않은 `TypeError`, `AttributeError`, `TimeoutError`가 외부로 직접 노출되지 않음
- 입력 검증 오류와 검색 파이프라인 오류가 명확히 구분됨
- 성공 결과는 항상 `recommendation`과 `source` 포함
- 실패 결과는 항상 `recommendation`과 `source`가 `None`
- 중복 추천이 발생하지 않음
